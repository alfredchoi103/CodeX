# Policy Mix Recommendation

Use this framework when the user wants actionable recommendations for government and related agencies, especially after a diagnostic analysis has already identified key problems.

## When to use

Use this framework when the goal is to move from diagnosis to action:

- recommending a policy mix
- assigning lead actors
- sequencing interventions
- balancing engineering, planning, governance, and behavioral tools
- preparing recommendations for a report, briefing, or thesis conclusion

This framework works especially well after `patching-packaging`, but it can also be used on its own.

## Core idea

A strong recommendation should not be a loose list of ideas. It should specify:

- policy objective
- tool type
- lead actor
- supporting actors
- time horizon
- implementation logic

## Recommendation design rules

### Rule 1: Mix, do not rely on one tool

Combine at least three tool families where relevant:

- substantive tools
- procedural or coordination tools
- implementation or calibration tools

### Rule 2: Match recommendation to actor capacity

Do not recommend tools that the relevant agencies cannot realistically deliver.

### Rule 3: Separate short-term patching from longer-term packaging

- short term: recalibration, pilots, hotspot action, protocol change
- medium term: standards revision, cross-department systems, institutional strengthening
- long term: statutory reform, governance redesign, strategic package reform

### Rule 4: Specify ownership

Every recommendation should name:

- one lead actor
- key support actors
- target group or implementation object

### Rule 5: Explain why the combination works

Show how tools complement each other rather than stand alone.

## Primary recommendation table template

| Policy problem | Recommended tool | Tool type | Lead actor | Support actors | Time horizon | Expected contribution |
| --- | --- | --- | --- | --- | --- | --- |
| identify the problem | state the recommendation | substantive, procedural, calibration, or hybrid | main bureau or department | related agencies | short, medium, long | what this tool helps achieve |

## Secondary sequencing table template

| Phase | Priority action | Why now | Dependencies | Main risk if delayed |
| --- | --- | --- | --- | --- |
| Short term | no-regret or urgent action | immediate exposure or feasibility | data, coordination, budget, mandate | worsening recurring losses |
| Medium term | system-strengthening reform | needed to consolidate early gains | interdepartmental agreement, standard updates | fragmented improvement |
| Long term | structural package reform | requires political preparation and legal change | legislation, capital programme, governance redesign | persistent lock-in |

## Suggested recommendation categories

When relevant, consider these categories:

- engineering and infrastructure
- land-use and planning control
- building and site-level standards
- warning, monitoring, and data integration
- emergency coordination
- public communication and behavior change
- finance, incentives, and maintenance
- institutional coordination and governance

## Applied guidance for Hong Kong inland flooding

For Hong Kong inland flooding, recommendations should usually combine:

| Category | Example recommendation direction | Likely actors |
| --- | --- | --- |
| Engineering | hotspot drainage upgrade, detention, micro-topography retrofit | DSD, Highways, Housing |
| Planning | runoff control in development approval, flood-sensitive zoning review | Planning Department, Development Bureau |
| Building standards | basement, podium, and entrance flood protection requirements | Buildings Department, Housing Department |
| Data and warning | integrate rainfall, drainage, transport, and district alerts | HKO, DSD, Transport Department, emergency services |
| Coordination | district-level flood action protocol with trigger thresholds | Security Bureau, District Offices, line departments |
| Long-term package | integrated urban stormwater resilience strategy | Development Bureau and cross-department steering body |

## Output pattern

When using this framework, structure the answer as:

1. `Framework chosen`
2. `Recommendation logic`
3. `Policy mix recommendation table`
4. `Sequencing`
5. `Advice to government and related departments`
